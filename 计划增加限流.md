# 计划增加限流 — 各模块循环内可配置间隔分析

> 分析日期：2026-06-19

## 一、背景

各模块在执行循环密集 RPC 请求时缺乏间隔，容易触发支付宝限流。需要在以下四个模块的循环迭代间添加统一可配置的延迟，时间由 `BaseModel.taskInterval` 控制（默认 1000ms，范围 0~10000ms），并在 Web 界面中显示。

## 三、当前代码状态

### 3.1 基础配置 (BaseModel.java)

- **当前工作区不包含** `taskInterval` 字段或 `sleepTaskInterval()` 方法
- 需新增：
  - `IntegerModelField taskInterval` — 范围 0~10000ms，默认 1000ms，显示名称"子任务循环间隔"
  - 静态方法 `sleepTaskInterval()` — 如果 `taskInterval > 0` 则延时该毫秒数
  - 在 `getFields()` 中注册以便显示到 UI

### 3.2 配置文件 [jim]-config_v2.json

- `BaseModel` 段已存在 `"taskInterval": {"value": 1000}`
- **但存在语法错误**：`toastOffsetY` 和 `taskInterval` 之间的逗号缺失，需要修复
- 其他四个模块的 `enable` 字段显示均为 `true`（已启用）

### 3.3 当前各模块已有延迟现状

当前各模块中使用的是**硬编码** `TimeUtil.sleep(xxx)`，值分散不统一：
- 200ms — 少量内部操作
- 300ms — ProtectEcology 大部分循环
- 500ms — AntOcean 任务领取 / AntMember 签到
- 1000ms — AntOcean/AntSports/AntMember 部分操作
- 2000ms — AntSports 任务完成等待

## 五、实施方案

### 5.1 BaseModel.java 修改

```java
// 新增字段（位置：在 toastOffsetY / enableOnGoing 之后）
@Getter
private static final IntegerModelField taskInterval = new IntegerModelField(
    "taskInterval", "子任务循环间隔(毫秒)", 1000, 0, 10000
);

// 新增静态辅助方法
public static void sleepTaskInterval() {
    int interval = taskInterval.getValue();
    if (interval > 0) {
        TimeUtil.sleep(interval);
    }
}
```

在 `getFields()` 中添加 `modelFields.addField(taskInterval);`

### 5.2 各模块修改策略

**策略一（推荐）**：所有循环迭代末尾统一调用 `BaseModel.sleepTaskInterval()`，将现有硬编码的 `TimeUtil.sleep(xxx)` 替换为该方法。

**策略二**：仅在当前没有延迟的循环添加，已有硬编码的保留不动（改动量较小但不够统一）。

**策略三**：仅添加主要循环（每模块 3-5 处），忽略嵌套过深或非关键路径上的小循环。

### 5.3 配置文件修复

`[jim]-config_v2.json` 中 `BaseModel` 段第 631-636 行，需在 `toastOffsetY` 对象后补上逗号：

```json
"toastOffsetY" : {
    "value" : 0
},                ← 缺少的逗号
"taskInterval" : {
    "value" : 1000
}
```

## 六、风险与注意事项

1. **设置为 0 则关闭延迟**：`IntegerModelField` 范围 0~10000，用户可自行关闭
2. **不影响非循环操作**：`sleepTaskInterval()` 仅在循环迭代间调用，不影响模块启动和顺序执行
3. **默认 1000ms 较保守**：对大多数用户足够，高频场景可调小（如 200ms）
4. **配置文件已有但代码缺失**：BaseModel 中缺少 `taskInterval` 字段，需先补充后代码才能识别该配置值
5. **与现有 sleep 不冲突**：一些业务逻辑本身需要 `TimeUtil.sleep(500)` 等待结果，这些应保留不动；仅替换人为限流的延迟
